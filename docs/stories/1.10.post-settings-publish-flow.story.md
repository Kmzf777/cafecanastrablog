## Story

**As a** blog administrator,
**I want** to configure post metadata and control publication,
**so that** each post is published with all the correct information.

## Acceptance Criteria

1. `PostSettings.tsx` implemented as the "Settings" tab in the editor sidebar, containing all post metadata fields
2. Fields: title (syncs with first H1 block if exists), slug (auto-generated from title, editable), featured image (upload with preview and remove), category (select from predefined list), tags (input with chips — add on Enter/comma, remove on click/backspace), author (input, default "Cafe Canastra"), locale (dropdown PT/EN/ES with flag indicators)
3. Locale selector with flags: when changing locale, check for existing translations via `translation_group_id` and show links to them
4. Link to create translation: "Create EN version" / "Create ES version" button that opens new editor with same `translation_group_id`
5. Reading time calculated and displayed automatically (readonly, from `reading-time.ts`)
6. Word count displayed automatically (readonly)
7. Three action buttons at bottom of sidebar:
   - **Save Draft** — saves with status `draft`
   - **Schedule** — opens date/time picker, saves with status `scheduled` and `scheduled_at`
   - **Publish Now** — saves with status `published` and `published_at = now()`
8. Before publishing: confirmation dialog showing "Publish now? SEO Score: XX/100"
9. If SEO score < 50: warning alert "SEO score is low. Do you still want to publish?" with Yes/Cancel
10. Full editor page layout at `/admin/posts/new/page.tsx` and `/admin/posts/[id]/edit/page.tsx`:
    - Left (70%): BlockEditor component
    - Right (30%): Sidebar with 3 tabs — SEO | GEO | Settings
    - Top: Header with Back button, post title, save/publish buttons
11. Both new and edit pages use the same editor layout, differing only in initial data (empty vs fetched)
12. After successful publish/save, redirect to `/admin` with success toast

## Integration Verification

- IV2: Scheduling works with existing cron endpoint (`/api/publish-scheduled`)
- IV3: Status transitions are correct: draft -> scheduled/published, scheduled -> published

### Key Technical Details
- **Existing PostForm.tsx**: Contains the full old form with textarea for content. The new editor layout replaces its usage entirely. The old component can be deleted or deprecated.
- **State management**: Use a single `useReducer` or multiple `useState` hooks in the page component. All child components receive state + callbacks.
- **Tag chips**: Implement with array state. On Enter or comma in input: add trimmed value, clear input. On chip X click: remove from array. Display as `<span>` badges.
- **Slug uniqueness**: On publish/save, the API should check slug uniqueness and return error if duplicate. Show error in toast.
- **Translation query**: On locale selector mount, query `SELECT id, locale, title FROM blog_posts WHERE translation_group_id = {id} AND id != {currentId}` to find sibling translations.
- **Tab component**: Use Radix `Tabs` from existing UI components for the sidebar.
- **Existing CATEGORIES**: `['Receitas', 'Terroir & Origem', 'Metodos de Preparo', 'Noticias', 'Curiosidades']` — defined in PostForm.tsx, extract to shared constant.
