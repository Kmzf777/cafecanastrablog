## Story

**As a** developer,
**I want** a complete data model supporting blocks, SEO config, and GEO config,

## Acceptance Criteria

1. TypeScript types created in `lib/types/blog.ts` (modified) and `lib/types/block-data.ts` (new): `BlockType` union type covering all 16 block types, `ContentBlock` interface with `id`, `type`, `order`, `data`, `settings`, `SeoConfig` interface, `GeoConfig` interface, `SemanticEntity` interface
2. All 16 block data interfaces created in `lib/types/block-data.ts`: `HeadingData`, `ParagraphData`, `ImageData`, `GalleryData`, `QuoteData`, `ListData`, `CodeData`, `EmbedData`, `DividerData`, `CalloutData`, `TableData`, `FaqData`, `CtaData`, `VideoData`, `AccordionData`, `ProductData`
3. Migration SQL `database/002_add_block_system.sql` created and validated:
   - New columns on `blog_posts`: `locale VARCHAR(5) DEFAULT 'pt'`, `translation_group_id UUID`, `seo_config JSONB DEFAULT '{}'`, `geo_config JSONB DEFAULT '{}'`, `reading_time_minutes INTEGER DEFAULT 0`, `word_count INTEGER DEFAULT 0`, `seo_score INTEGER DEFAULT 0`
   - New table `blog_post_blocks` with: `id UUID PK`, `post_id UUID FK CASCADE`, `type VARCHAR(30) CHECK`, `order INTEGER NOT NULL`, `data JSONB NOT NULL`, `settings JSONB`, `created_at`, `updated_at`
   - RLS policies: public read for blocks of published posts, no public writes
   - Indices: `idx_blocks_post_id`, `idx_blocks_post_order`, `idx_blocks_type`, `idx_blog_posts_translation_group`, `idx_blog_posts_locale`
   - Trigger for `updated_at` on `blog_post_blocks`
4. Migration SQL `database/003_migrate_content.sql` created: converts existing posts' `content` field into a single `paragraph` block in `blog_post_blocks`, preserving original Markdown content
5. Zod validation schemas created in `lib/schemas/block-validation.ts` for all 16 block types, each validating the `data` JSONB structure
6. `npm run typecheck` passes with zero errors
7. Existing `BlogPost` interface updated to include new fields while maintaining backward compatibility (new fields optional)

## Integration Verification
- IV1: Existing posts remain accessible via current public API after migration SQL is applied
- IV2: RLS policies work correctly — public can read blocks of published posts, cannot write
- IV3: Query performance: SELECT post + JOIN blocks < 200ms with 50 posts and 500 blocks

### Key Technical Details
- **Supabase PostgreSQL** — all SQL must be compatible with Supabase SQL Editor
- Existing `blog_posts` table schema is in `database/001_create_blog_posts.sql`
- Current `BlogPost` type: `id, title, slug, content, excerpt, image_url, image_alt, status, scheduled_at, published_at, author, category, tags, created_at, updated_at`
- New fields MUST be optional (nullable or with defaults) to avoid breaking existing queries
- `translation_group_id` defaults to `gen_random_uuid()` — for existing posts, migration sets it to the post's own `id`
- Block `type` CHECK constraint must include all 16 types: heading, paragraph, image, gallery, quote, list, code, embed, divider, callout, table, faq, cta, video, accordion, product
- Block `data` JSONB structure varies by type — see PRD Section 6.1 for complete interface definitions
- Block `settings` JSONB: `{ width, alignment, css_class, spacing }`
- Zod uses `z.discriminatedUnion` on `type` field for block data validation
