## Story
**As a** desenvolvedor do projeto,
**I want** criar a tabela `blog_posts` no Supabase com todas as colunas necessárias, índices, trigger de `updated_at`, RLS habilitado, e o bucket `blog-images` no Storage com políticas adequadas,
**so that** o sistema de blog tenha uma base de dados robusta e segura para armazenar posts e imagens.

## Acceptance Criteria

1. A tabela `blog_posts` existe no Supabase com as seguintes colunas:
   - `id` UUID PRIMARY KEY DEFAULT gen_random_uuid()
   - `title` VARCHAR(255) NOT NULL
   - `slug` VARCHAR(255) NOT NULL UNIQUE
   - `content` TEXT NOT NULL
   - `excerpt` TEXT (nullable)
   - `image_url` TEXT (nullable)
   - `image_alt` TEXT (nullable)
   - `status` VARCHAR(20) DEFAULT 'draft' CHECK (status IN ('draft', 'scheduled', 'published'))
   - `scheduled_at` TIMESTAMPTZ (nullable)
   - `published_at` TIMESTAMPTZ (nullable)
   - `author` VARCHAR(100) DEFAULT 'Café Canastra'
   - `category` VARCHAR(100) (nullable)
   - `tags` TEXT[] DEFAULT '{}'
   - `created_at` TIMESTAMPTZ DEFAULT NOW() NOT NULL
   - `updated_at` TIMESTAMPTZ DEFAULT NOW() NOT NULL

2. A tabela possui os seguintes índices:
   - `idx_blog_posts_slug` em `slug`
   - `idx_blog_posts_status` em `status`
   - `idx_blog_posts_scheduled_at` em `scheduled_at` (para query do cron)
   - `idx_blog_posts_published_at` em `published_at DESC` (para ordenação da listagem pública)

3. Existe um trigger `update_blog_posts_updated_at` que atualiza `updated_at = NOW()` automaticamente em qualquer UPDATE na tabela.

4. RLS (Row Level Security) está habilitado na tabela com as seguintes políticas:
   - Leitura pública: qualquer usuário pode ler posts com `status = 'published'`
   - Escrita restrita: apenas `service_role` pode INSERT, UPDATE, DELETE

5. O bucket `blog-images` existe no Supabase Storage com:
   - Acesso público para leitura (leitura de imagens sem autenticação)
   - Limite de tamanho de arquivo: 5MB
   - MIME types permitidos: `image/jpeg`, `image/png`, `image/webp`

6. Um arquivo SQL de migração `database/001_create_blog_posts.sql` é criado no projeto com todo o DDL executado, para documentação e replicação futura.

7. Uma função Supabase helper `get_published_posts()` opcional não é necessária — as queries são feitas diretamente via SDK.

8. Os tipos TypeScript da tabela são criados em `lib/types/blog.ts` para uso nas APIs e componentes.

## Dev Notes

### DDL Completo para Executar no Supabase

```sql
-- Tabela principal
CREATE TABLE public.blog_posts (
  id           UUID DEFAULT gen_random_uuid() PRIMARY KEY,
  title        VARCHAR(255) NOT NULL,
  slug         VARCHAR(255) NOT NULL UNIQUE,
  content      TEXT NOT NULL,
  excerpt      TEXT,
  image_url    TEXT,
  image_alt    TEXT,
  status       VARCHAR(20) DEFAULT 'draft'
               CHECK (status IN ('draft', 'scheduled', 'published')),
  scheduled_at TIMESTAMPTZ,
  published_at TIMESTAMPTZ,
  author       VARCHAR(100) DEFAULT 'Café Canastra',
  category     VARCHAR(100),
  tags         TEXT[] DEFAULT '{}',
  created_at   TIMESTAMPTZ DEFAULT NOW() NOT NULL,
  updated_at   TIMESTAMPTZ DEFAULT NOW() NOT NULL
);

-- Índices
CREATE INDEX idx_blog_posts_slug         ON public.blog_posts(slug);
CREATE INDEX idx_blog_posts_status       ON public.blog_posts(status);
CREATE INDEX idx_blog_posts_scheduled_at ON public.blog_posts(scheduled_at)
  WHERE status = 'scheduled';
CREATE INDEX idx_blog_posts_published_at ON public.blog_posts(published_at DESC)
  WHERE status = 'published';

-- Trigger updated_at
CREATE OR REPLACE FUNCTION update_blog_posts_updated_at()
RETURNS TRIGGER AS $$
BEGIN
  NEW.updated_at = NOW();
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_blog_posts_updated_at
  BEFORE UPDATE ON public.blog_posts
  FOR EACH ROW
  EXECUTE FUNCTION update_blog_posts_updated_at();

-- RLS
ALTER TABLE public.blog_posts ENABLE ROW LEVEL SECURITY;

-- Política: qualquer um pode ler posts publicados
CREATE POLICY "Public can read published posts"
  ON public.blog_posts
  FOR SELECT
  USING (status = 'published');

-- Política: service_role pode fazer tudo (bypass RLS automático no Supabase)
-- O service_role já tem bypass por padrão — mas para INSERT/UPDATE/DELETE via anon,
-- criar policy restritiva:
CREATE POLICY "No public writes"
  ON public.blog_posts
  FOR ALL
  USING (false)
  WITH CHECK (false);
```

```typescript
export interface BlogPost {
  id: string

  excerpt: string | null
  image_url: string | null
  image_alt: string | null
  status: BlogPostStatus
  scheduled_at: string | null

  author: string
  category: string | null
  tags: string[]
  created_at: string
  updated_at: string
}

export type CreateBlogPost = Omit<BlogPost, 'id' | 'created_at' | 'updated_at'>

```

### Supabase Storage — Bucket `blog-images`
- Criar como público no dashboard: Storage → Create bucket → Nome: `blog-images` → Public: ON
- URLs das imagens seguirão o padrão: `https://[supabase-project].supabase.co/storage/v1/object/public/blog-images/[filename]`

### Atenção: service_role vs anon key
- A API de admin (criação/edição de posts) deve usar `SUPABASE_SERVICE_ROLE_KEY` no backend
- O site público (listagem de posts) pode usar `NEXT_PUBLIC_SUPABASE_ANON_KEY` com RLS
- O script de cron de publicação deve usar `SUPABASE_SERVICE_ROLE_KEY`
